# AURUS Control Panel — экран «Игроки»

## 1. Назначение документа

Этот документ описывает итоговую концепцию экрана **«Игроки»** для будущего приложения **AURUS Control Panel**.

Документ предназначен для ИИ-агента, который будет помогать в разработке проекта в VS Code.

Главная задача экрана **«Игроки»** — навести порядок в составе альянса AURUS [SILA]:

- кто является реальным участником альянса;
- какими аккаунтами управляет каждый человек;
- какие расы у этих аккаунтов;
- кто активен, кто новичок, кто спящий;
- кто заполнил анкету, а по кому данных ещё нет;
- кто готов помогать альянсу;
- у кого есть спорные или неподтверждённые связи аккаунтов;
- какие роли и направления закреплены за игроками.

Ключевой принцип: **на экране «Игроки» показываются люди, а не отдельные аккаунты**.

Аккаунты раскрываются внутри карточки конкретного игрока.

---

## 2. Контекст проекта

Мы разрабатываем систему управления альянсом **AURUS [SILA]** в браузерной космической онлайн-игре **XCraft**.

Альянс старый, создан **11.03.2019 игроком DREDGV**.

Сейчас в альянсе много участников, включая старых игроков, новичков и игроков с несколькими аккаунтами разных рас. Один человек может управлять аккаунтами терранов, жуков/зергов и тоссов/протоссов. Это создаёт путаницу: не всегда понятно, кто есть кто, кому принадлежит конкретный аккаунт и чем этот игрок может быть полезен альянсу.

Поэтому первый важный модуль приложения — не карта, не флот и не станции, а именно **учёт людей и аккаунтов**.

---

## 3. Главный принцип структуры данных

Нельзя строить систему по принципу:

```text
Игрок = аккаунт
```

Правильная модель:

```text
Человек / участник альянса
    ├── Аккаунт 1: терран
    ├── Аккаунт 2: жук / зерг
    ├── Аккаунт 3: тосс / протосс
    ├── Флот по каждому аккаунту
    ├── Объекты по каждому аккаунту
    ├── Роли и задачи человека
    └── Комментарии и статус проверки
```

Основная сущность — **человек / участник альянса**.

Аккаунт — это отдельная сущность, связанная с человеком.

Один человек может иметь несколько аккаунтов.

Один аккаунт должен иметь одного основного владельца, но статус связи может быть разным: подтверждён, не подтверждён, предположительно, спорно и т.д.

## 4. Назначение экрана «Игроки»

Экран должен быстро отвечать на вопросы:

1. Кто этот человек?
2. Как его зовут и как к нему обращаться?
3. Какие аккаунты он ведёт?
4. Какими расами он играет?
5. Какой аккаунт у него основной?
6. Активен ли он?
7. Заполнил ли он анкету?
8. Всё ли понятно по его аккаунтам?
9. Чем он занимается в альянсе?
10. Готов ли он помогать?
11. Нужна ли ему помощь?
12. Есть ли внутренние комментарии руководства?

Экран должен быть **центром идентификации состава альянса**.

## 7. Верхняя панель статистики

На экране должны быть не декоративные, а управленческие показатели.

Рекомендуемые карточки:

| Карточка | Значение |
|---|---|
| Всего игроков | Сколько людей заведено в базе |
| Активные | Кто реально играет / отвечает |
| Без анкеты | Кто ещё не прислал данные |
| Неясные аккаунты | Сколько аккаунтов без подтверждённого владельца |
| Готовы помогать | Потенциальный актив альянса |
| Нужна помощь | Игроки, которым нужна поддержка |

Пример:

```text
Всего игроков: 85
Активные: 63
Без анкеты: 31
Неясные аккаунты: 14
Готовы помогать: 18
Нужна помощь: 9
```

## 8. Основная таблица игроков

Главная таблица должна показывать **людей**, а не отдельные аккаунты.

| Колонка | Назначение |

| Анкета | Заполнена / частично / нет / нужно обновить |
| Аккаунты | Количество связанных аккаунтов |
| Расы | Терран / Жук / Тосс |

| Роль | Игрок, актив, куратор, Совет, зам, глава |

| Активность | Онлайн / недавно / редко / давно нет |
| Проверка | Всё ясно / уточнить / есть вопросы |
| Действия | Открыть карточку / редактировать |

| Zuka | Актив | ✅ | 3 | Т / Ж / П | Zuka | Совет | ОПС / астероиды | Онлайн | ✅ ясно |
| 1917 | Актив | 🟡 | 2 | Т / П | 1917 | Офицер | логистика | Недавно | 🟡 уточнить |
| PlayerX | Новичок | ❌ | 1 | Ж | PlayerX | Игрок | развитие | Редко | 🔴 нет данных |

## 9. Обозначение рас

Для компактного отображения можно использовать короткие обозначения:

```text
Т = терран
Ж = жук / зерг
П = тосс / протосс
```

Цветовая логика:

```text
Терран — голубой / синий
Жук / зерг — красный / оранжевый
Тосс / протосс — фиолетовый / золотой
```

В таблице можно показывать как:

```text
[Т] [Ж] [П]
```

или:

```text
Т / Ж / П
```

```text
3. Быстрые фильтры:
   - без анкеты;
   - неясные аккаунты;
   - готовые помогать;
   - несколько рас.

4. Боковую карточку выбранного игрока.

5. Вкладку аккаунтов внутри полной карточки.

6. Статусы подтверждения связей аккаунтов.
```

---

## 28. Чего не делать в первой версии

Не нужно сразу реализовывать:

```text
сложную карту;
визуализацию зон видимости;
альянсовые станции и бонусы;
полноценный журнал аудита;
сложные права доступа;
аналитику флота;
графы связей.
```

Эти модули важны, но они должны идти после наведения порядка в данных игроков и аккаунтов.

---

## 29. Требования к ИИ-агенту

ИИ-агент должен:

1. Сохранять принцип: **человек отдельно, аккаунты отдельно**.
2. Не смешивать игрока и аккаунт в одну сущность.
3. Не считать неподтверждённые связи фактами.
4. Предусмотреть статусы проверки.
5. Делать структуру расширяемой.
6. Не добавлять лишние игровые механики без подтверждения пользователя.
7. Сначала проектировать данные, потом интерфейс.
8. Не удалять существующие данные.
9. Перед изменениями делать резервные копии файлов.
10. Объяснять, какие поля и таблицы были добавлены.

---

## 30. Итоговое понимание

Экран **«Игроки»** должен быть не просто списком состава, а центром наведения порядка в альянсе.

Главная логика:

```text
Человек → аккаунты → расы → активность → роль → польза → проблемы
```

До тех пор, пока не наведён порядок в людях и аккаунтах, остальные модули — карта, станции, видимость, бонусы, флот — будут строиться на неустойчивой основе.

Поэтому модуль **«Игроки»** является первым ключевым модулем будущего приложения **AURUS Control Panel**.
